**Datum:** 2026-04-18

## Syfte

Flytta `anthropic_api_key` från klartext i `settings.json` (som idag ligger i `%APPDATA%/svoice-v3/settings.json`) till Windows Credential Manager via `keyring`-crate. Förbered namespace för framtida Google OAuth-tokens (iter 4).

Ger:
- Säkerhet: nyckeln krypteras vid vila via Windows DPAPI.
- Distribution: samma `settings.json`-fil kan synas eller delas utan att läcka nyckel.
- Future-proof: samma mekanism används för OAuth-tokens i iter 4.

## Icke-mål

- Ingen UI-motsvarighet för OAuth (det är iter 4).
- Ingen browser-integration / SSO-flöde.
- Ingen migrering bort från `settings.json` för icke-känsliga inställningar — filen är fortsatt source of truth för resten.

## Arkitektur

### Ny crate: `svoice-secrets`

Pure library i `src-tauri/crates/secrets/`, inga Tauri-deps. Wrappar `keyring = "3"`.

Namespace-konvention:
- `service = "svoice-v3"`
- `username` = semantisk nyckel, t.ex. `"anthropic_api_key"`, `"google_access_token"`, `"google_refresh_token"`.

Alla secrets samlas under service `"svoice-v3"` i Credential Manager — enkel att söka och rensa vid ominstallation.

### Publikt API

```rust
// src-tauri/crates/secrets/src/lib.rs
pub const SERVICE: &str = "svoice-v3";

#[derive(thiserror::Error, Debug)]
pub enum SecretsError {
    #[error("keyring backend unavailable: {0}")]
    Backend(#[from] keyring::Error),
}

pub fn get_anthropic_key() -> Result<Option<String>, SecretsError>;
pub fn set_anthropic_key(key: &str) -> Result<(), SecretsError>;
pub fn delete_anthropic_key() -> Result<(), SecretsError>;
pub fn has_anthropic_key() -> bool; // swallow errors → false
```

Semantik:
- `get` returnerar `Ok(None)` när entry saknas (normalfall vid first-run) och `Err` endast vid backend-fel (t.ex. roaming-profile-sync-issue).
- `has` är convenience för frontend; får aldrig blocka eller krascha.
- Mot `keyring::Entry::get_password` mappas `keyring::Error::NoEntry` till `Ok(None)`, övriga till `Err(Backend)`.

Tre nya Tauri-commands i `src-tauri/crates/ipc/src/commands.rs`:

| Command | Signatur | Syfte |
|---|---|---|
| `has_anthropic_key` | `() -> bool` | Settings.tsx visar `••••••••` eller tom input |

```ts

```

## Migration från klartext-JSON

Vid app-start (i `setup`-closure i `src-tauri/src/lib.rs`, före worker-spawn):

```rust
migrate_anthropic_key_from_json()?;
```

Funktionen placeras i `src-tauri/src/lib.rs` (app-level orchestration) — varken `svoice-settings` eller `svoice-secrets` behöver depa på den andra. Logiken:
1. Läser `%APPDATA%/svoice-v3/settings.json` som `serde_json::Value`.
2. Kolla om `anthropic_api_key` finns som non-empty string.
3. Om ja: kör `svoice_secrets::set_anthropic_key(&value)`. Vid lyckat resultat, ta bort fältet ur JSON-objektet och skriv tillbaka atomiskt (tempfil + rename).
4. Vid keyring-fel: logga `tracing::error`, lämna klartext kvar. Migrering försöker igen nästa start.

Best-effort; idempotent (körs varje start men är no-op om fältet redan är borta).

**Ingen user-notis** — migration är tyst. User ser bara att fältet i Settings visar `••••••••` efter första start.

## Frontend-UX

`src/windows/Settings.tsx`, API-key-sektionen (rad 507–531):

- Vid mount: `hasAnthropicKey()` → setStateLocal. Om `true`, input placeholder `"••••••••"` och `value=""`.
- Vid user-input: `draft`-state får nytt fält `newAnthropicKey: string | null` (null = orört, tom = raderat, icke-tomt = ny key).
- Vid "Spara"-klick:
  - Om `newAnthropicKey !== null && newAnthropicKey !== ""`: `setAnthropicKey(newAnthropicKey)`.
  - Om `newAnthropicKey === ""`: `clearAnthropicKey()`.
  - Om `null`: ingen key-action.
- **Rensa-knapp:** liten länk `Rensa nyckel` under input-fältet, synlig endast om `hasAnthropicKey === true && newAnthropicKey === null`. Klick sätter `newAnthropicKey = ""` (rendras som tom input + visar hint "Trycker Spara så rensas nyckeln").
- **Hjälptext byts:** `"Sparas säkert i Windows Credential Manager."` (nuvarande text *"Sparas i klartext — keyring kommer i nästa iter."* försvinner).

## Error-hantering

Fail-soft, inga klartext-fallbacks.

- `get_anthropic_key()` backend-fel → `llm = None` → action-popup visar befintligt felmeddelande *"Ingen LLM-nyckel konfigurerad..."*.
- `set_anthropic_key()`/`clear_anthropic_key()` IPC-fel → returnerar `Err(String)`. Settings.tsx visar inline-error i API-key-sektionen.
- `tracing::error`-log i alla backend-felbranch så `cargo tauri dev`-konsol kan användas för diagnostik.

## Risk & rollback

- **Risk:** `keyring::Entry::new()` kan returnera fel på corrupt user-profile. Mitigering: fail-soft; ingen nyckel = action-popup disabled med tydligt error. User kan köra app, bara inte Claude-provider.
- **Rollback:** revertera merge-commit. Befintliga keyring-entries kvar i Credential Manager men inte används — ej skadligt, user kan rensa manuellt.
